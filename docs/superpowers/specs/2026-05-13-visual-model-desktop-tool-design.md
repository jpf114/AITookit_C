# 视觉模型桌面工具设计文档

## 1. 目标

构建一个面向终端用户的 Windows 桌面应用，通过图形界面在本地运行轻量视觉模型。第一版聚焦于基于 ONNX 的目标检测，支持本地图像或视频输入、可视化结果叠加，以及结果导出。整个应用应呈现专业桌面工具的气质，而不是偏工程调试台的形态。

## 2. 产品定位

这是一个面向普通用户的视觉推理工具。用户无需理解底层推理框架和张量细节，只需要加载模型、选择图片或视频、执行推理、查看结果、导出输出即可。

第一阶段明确不做以下方向：

- 不做训练平台
- 不做插件市场
- 不做通用多模态 AI 外壳

第一阶段范围严格收敛为：

- 仅支持 Windows 桌面
- 仅支持轻量视觉模型
- 仅支持本地推理
- 仅支持目标检测任务
- 仅支持 ONNX 作为推理后端

这样的范围足够小，便于快速做出能用的第一版，同时也为后续扩展分类、分割、OCR、人脸分析和多后端运行时保留清晰路径。

## 3. 用户与核心使用流程

### 主要用户

主要用户是非开发人员，或只有轻度技术背景的业务操作者。他们希望通过一个清晰的桌面界面直接使用本地模型，而不是面对一堆运行参数和开发概念。

### 核心流程

第一版优化的主流程如下：

1. 打开应用
2. 选择模型包或 ONNX 文件
3. 选择输入图片、图片文件夹或视频
4. 调整基础阈值
5. 执行推理
6. 查看叠加后的检测结果
7. 导出渲染结果或结构化结果数据

界面应该尽量减少术语，避免用户被输入张量、预处理、运行时后端这些细节打断。

## 4. UI 风格方向

界面风格和交互密度参考现有项目 `D:\Code\MyProject\GIS_TOOL` 的桌面语言。

需要继承的特征包括：

- 专业桌面软件气质
- 左侧导航结构
- 结构化工作区，而不是营销页式布局
- 适中的信息密度
- 统一的图标和样式表管理方式
- 清晰的状态面板，而不是大量悬浮式装饰组件

本项目不需要机械地复制 `GIS_TOOL` 的页面内容，因为两者的核心工作区不同。这个 AI 工具的中心体验应是图像预览和推理结果叠加，周边面板用于支撑这件事。

## 5. 主界面布局

第一版主界面采用基于 `QMainWindow` 和分栏器的三段式桌面布局。

### 左侧导航栏

左侧导航提供稳定的一级入口：

- 首页
- 模型
- 推理
- 结果
- 设置

导航栏应保持紧凑，视觉风格尽量贴近 `GIS_TOOL` 的既有模式。

### 中央工作区

中央区域是整个应用的视觉重心，应始终承担主要任务展示。

- 在首页，显示最近使用项、快捷入口和常用模型
- 在推理页，显示当前图片或视频帧
- 在结果页，显示渲染后的结果图和导出相关摘要

目标检测结果应直接绘制在预览区域，包括检测框、类别标签和置信度。

### 右侧上下文面板

右侧面板根据当前页面和状态动态变化：

- 模型页：显示模型元数据、任务类型、标签、输入尺寸、默认阈值
- 推理页：显示输入源控制、置信度阈值、NMS 阈值、运行按钮
- 结果页：显示结果摘要、导出按钮、目标数量、输出选项
- 设置页：显示运行偏好、路径、默认值等设置

第一版应尽量避免过多模态对话框，主要操作应留在主窗口内完成。

## 6. 第一版功能范围

第一版只实现让目标检测真正可用的最小闭环。

### 包含内容

- 加载基于 ONNX 的目标检测模型
- 通过清单文件加载模型包
- 支持通过引导流程加载裸 ONNX 文件
- 打开单张图片
- 打开图片文件夹并进行批量处理
- 打开视频文件
- 执行本地目标检测
- 在预览区域绘制检测框和标签
- 导出渲染后的图片结果
- 导出 JSON 结构化结果
- 保存并恢复最近使用的模型和输入

### 不包含内容

- 模型训练
- 在线模型下载
- 远程推理
- 在 UI 中切换多推理后端
- 插件市场式能力管理
- 分割、OCR、分类、人脸类能力
- 复杂任务中心或长任务编排系统

这些排除项是有意为之，在第一版检测工作流真正稳定之前不应提前加入。

## 7. 技术栈

第一阶段技术栈固定如下：

- 语言：C++17
- 界面：Qt Widgets
- 构建系统：CMake
- 依赖管理：使用本地共享安装的 vcpkg
- 图像处理辅助：OpenCV
- 推理后端：ONNX Runtime
- 测试：GoogleTest

第一版优先选择 Qt Widgets，而不是 QML。原因是当前项目更需要稳定、易调试、偏工具型的桌面界面，也更方便和 `GIS_TOOL` 的工程方式保持一致。

## 8. 架构设计

代码结构应按职责拆分为若干清晰模块，避免将 UI、推理、模型解释混在同一批文件里。

### `src/app`

负责应用启动、全局初始化、应用元信息、样式表加载、字体设置和启动流程装配。

### `src/ui`

负责 Qt 窗口、页面、控件布局和展示逻辑，不应直接编写模型推理实现。

建议包含的子区域：

- `main_window`
- `nav`
- `pages/home`
- `pages/models`
- `pages/inference`
- `pages/results`
- `pages/settings`
- 若干可复用控件，如预览控件、参数卡片、状态栏等

### `src/core`

提供跨模块共用的基础类型和通用设施。

建议职责包括：

- 图像和视频输入源描述
- 应用设置模型
- 结果数据结构
- 错误与状态类型
- 日志辅助
- 路径辅助
- JSON 辅助

### `src/runtime`

负责底层推理后端抽象。该层只处理模型运行与张量交换，不理解检测语义。

建议职责包括：

- 推理后端接口
- ONNX Runtime 实现
- 模型会话生命周期管理
- 输入张量准备辅助
- 原始输出张量获取

### `src/models`

负责任务级模型适配。该层理解模型语义，并将原始输出转换成可用的检测结果。

建议职责包括：

- 视觉模型统一接口
- 目标检测模型实现
- 预处理策略
- 后处理策略
- 标签映射
- 不同输出格式解码器

### `src/services`

负责把 UI 动作和底层模型能力串起来，作为界面与推理逻辑之间的中间层。

建议职责包括：

- 模型加载流程
- 单图推理流程
- 批量推理流程
- 视频推理流程
- 导出流程
- 最近使用项管理

### `resources`

存放图标、样式、示例模型清单、示例配置等静态资源。

### `tests`

存放核心逻辑、模型适配、运行时封装和服务流程相关的单元测试与集成测试。

## 9. 核心抽象边界

系统必须明确区分“运行模型”和“解释模型输出”这两件事。

### 推理后端抽象

定义一个后端接口，至少包含以下操作：

- 加载模型会话
- 接收准备好的输入张量
- 执行推理
- 返回原始输出张量

这一层只负责“把模型跑起来”，不负责理解输出含义。

### 业务模型抽象

定义一个更高层的视觉模型接口，至少包含以下操作：

- 加载模型包信息
- 校验模型所需资源
- 对图像执行推理
- 返回结构化检测结果
- 报告模型支持能力

这一层负责图像预处理、输出解码、置信度过滤、NMS 和标签映射。

这个边界必须严格保持，不能把运行时逻辑和检测后处理塌缩进一个大类里。

## 10. 模型包格式

推荐的模型加载方式是通过清单文件，例如 `model.json`，描述一个完整模型包。

模型包需要提供足够元数据，使应用在运行模型时不需要反复向用户追问底层细节。

### 清单中的必需字段

- 模型显示名称
- 任务类型
- 后端类型
- 模型文件路径
- 输入宽高
- 输入通道顺序
- 归一化参数
- 标签文件路径或内嵌标签列表
- 默认置信度阈值
- 默认 NMS 阈值
- 输出解码类型

### 推荐模型包结构

模型包应当是一个目录或一组关联文件，而不是只有一个裸模型文件。例如：

- `model.json`
- `model.onnx`
- `labels.txt`
- `README.md`

### 裸 ONNX 兼容流程

UI 也支持直接选择 `.onnx` 文件。此时应用应弹出一个简化配置流程，让用户补齐最低限度元数据，并提供保存为模型包清单的能力，便于后续复用。

整体上，推荐用户优先使用清单式模型包，因为它的使用体验更稳定。

## 11. 输入类型

第一版支持以下三种输入源：

- 单张图片
- 图片文件夹
- 视频文件

这三类输入应在内部映射到统一的输入源描述结构，避免 UI 和服务层到处写分支逻辑。

第一版不支持实时摄像头输入。

## 12. 输出与结果数据模型

第一版需要定义一套统一的目标检测结果结构，用于结果绘制和结果导出。

每个检测目标至少包含：

- 类别 ID
- 类别名称
- 置信度
- 边界框
- 渲染颜色

每次推理结果至少包含：

- 来源文件或帧标识
- 图像尺寸
- 推理耗时
- 目标总数
- 检测结果列表

这套结果模型应在 UI 渲染和 JSON 导出之间复用，避免重复转换。

## 13. 错误处理与用户反馈

第一版应优先提供明确、可读的错误提示，而不是静默失败。

需要覆盖的错误类别包括：

- 模型包无效
- 模型文件或标签文件缺失
- 输出解码类型不支持
- 输入文件不可读
- 运行时加载失败
- 推理执行失败
- 导出路径失败

UI 反馈应遵守以下原则：

- 在用户操作附近给出即时校验反馈
- 技术细节可以提供，但不要一开始就把底层错误直接砸给用户
- 除非用于排障，否则尽量避免直接暴露后端术语
- 出错后尽量保留当前窗口状态，避免用户重新走完整流程

对批量推理和视频推理这类耗时任务，应提供进度反馈；如果实现成本可控，第一版也应预留取消能力。

## 14. 配置持久化

应用需要保存轻量本地配置，包括：

- 最近使用模型
- 最近输入路径
- 最近导出目录
- 针对模型保存的最近阈值

持久化层保持简单即可，第一版可以使用 Qt Settings 或本地 JSON 文件。

## 15. 构建与依赖约定

构建组织方式参考 `D:\Code\MyProject\GIS_TOOL`。

### 构建规则

- 仓库根目录使用 `CMakeLists.txt`
- 使用 `CMakePresets.json` 提供标准开发入口
- 通过环境变量 `VCPKG_ROOT` 定位本地共享 vcpkg
- 所有依赖通过 CMake target 链接，不使用硬编码库路径

### 默认构建目录

- `build/debug`
- `build/release`

### 安装规则

- 默认安装前缀为 `install`
- 默认交付目标为 `Release`
- `Debug` 保留给本地开发和调试

### VS Code 支持

仓库中应生成或维护 `.vscode` 目录，至少提供以下内容：

- CMake 配置与构建任务
- GUI 程序调试启动配置
- 与 CMake Presets 对齐的工作区设置

目标是让该项目在本机上使用 VS Code 打开后，可以直接沿着 `GIS_TOOL` 的习惯开发，而不需要额外做大量编辑器配置。

## 16. 第一阶段依赖集合

第一阶段依赖应尽量精简，建议包含：

- Qt Widgets 相关能力
- Qt SVG 图标支持
- OpenCV
- ONNX Runtime
- GoogleTest

如果本地 vcpkg 端口布局需要少量适配，也应继续保持这套依赖管理思路，而不是转向手工散装集成。

## 17. 安装输出预期

安装后的 `Release` 结果应尽量接近“可直接运行的应用”，而不是只复制一个 exe。

安装内容至少应包括：

- 应用主程序
- Qt 运行时库
- Qt 平台插件文件
- OpenCV 运行时库
- ONNX Runtime 运行时库
- 图标与样式资源
- 示例模型清单或示例资源（如提供）

项目应自动化处理运行时依赖复制，整体思路参考 `GIS_TOOL` 当前的做法。

## 18. 测试策略

第一阶段测试重点放在稳定逻辑和高价值链路上。

### 单元测试

- 模型清单解析
- 设置项序列化
- 检测结果结构
- 预处理参数校验
- 后处理与输出解码逻辑
- NMS 行为

### 集成测试

- 模型包加载
- ONNX Runtime 会话启动
- 单图推理服务流程
- JSON 导出流程

### GUI 验证

第一阶段不要求重度 GUI 自动化，但应至少有明确的手工验证标准，覆盖：

- 加载模型包
- 打开图片
- 执行检测
- 查看检测框叠加效果
- 导出结果

## 19. 后续扩展路径

架构应为后续扩展保留空间，但不在第一阶段提前实现：

- 分类、分割等其他任务类型
- NCNN、MNN、TensorRT 等其他运行时
- 更丰富的结果对比能力
- 批任务管理
- 摄像头输入

这些都是后续方向，不属于当前范围。第一阶段不应为“未来也许会有的功能”做过度设计。

## 20. 第一阶段验收标准

当以下条件全部满足时，可认为第一阶段达标：

- 应用可以在 Windows 上通过 CMake、Presets 和本地 vcpkg 正常构建
- 工程提供独立的 Debug 和 Release 构建目录
- Release 安装结果可运行
- UI 风格符合 `GIS_TOOL` 的桌面工具参考方向
- 用户可以加载目标检测模型包
- 用户可以对图片执行目标检测
- 检测结果能清晰绘制在预览区域
- 用户可以导出渲染图和 JSON 结果
- 架构上已明确分离 UI、服务层、运行时和模型适配层

## 21. 范围护栏

实现过程中应明确遵守以下边界：

- 不把项目做成泛化的 AI 大平台
- 不加入训练流程
- 不把运行时后端代码写进 UI 类
- 不把 `lite.ai.toolkit` 绑定为产品底座
- 不同时启动多个任务类型
- 不在第一版检测工作流未稳定前提前搭大型插件系统

## 22. 总结

本项目第一版是一个基于 Qt Widgets、CMake、vcpkg、OpenCV 和 ONNX Runtime 的 Windows 桌面目标检测工具。它的界面风格参考 `GIS_TOOL` 的桌面语言，但交互重心放在图像预览和推理结果可视化上。整体架构刻意分离应用外壳、UI、服务层、运行时后端和模型适配层，以便在后续扩展时仍然保持结构清晰，而不会长成一个难以维护的一次性工程。
